# DocMan Component Library

Welcome to the **DocMan Component Library** documentation! This Storybook contains all the reusable components, patterns, and design guidelines for the Document Management System.

## üéØ Purpose

This component library serves as:
- **Single Source of Truth** for all UI components
- **Development Reference** for consistent implementation
- **Design System Documentation** for maintaining visual consistency
- **Testing Environment** for component behavior and interactions

## üìö What's Included

### Shared Components
Our shared component library includes:

- **BaseModal** - Foundation for all modal dialogs
- **FormField** - Standardized form inputs with validation
- **DataTable** - Sortable, filterable data tables
- **LoadingSpinner** - Consistent loading states
- **ErrorBoundary** - Graceful error handling

### Custom Hooks
Reusable logic patterns:

- **useStakeholderManagement** - Multi-select stakeholder logic
- **useExternalContacts** - Contact management
- **useReviewManagement** - Review workflow state
- **useFormData** - Common form data loading
- **useFileUpload** - Upload state & progress
- **useDocument** - Document loading & management
- **useUserRole** - Authentication & permissions
- **useApi** - API operations with loading states

## üé® Design Principles

### Consistency
All components follow consistent:
- **Color schemes** using Tailwind CSS
- **Typography** with standardized font sizes and weights
- **Spacing** using consistent margin and padding scales
- **Interactive states** for hover, focus, and disabled states

### Accessibility
Components are built with accessibility in mind:
- **Keyboard navigation** support
- **Screen reader** compatibility
- **ARIA labels** and roles
- **Color contrast** compliance
- **Focus management** for modals and forms

### Performance
Optimized for performance:
- **Lazy loading** where appropriate
- **Memoization** for expensive operations
- **Efficient re-renders** with proper dependency arrays
- **Bundle size optimization** with tree shaking

## üöÄ Getting Started

### Using Components

```jsx
import { BaseModal, FormField, DataTable } from '@/components/shared';

function MyComponent() {
  return (
    <BaseModal isOpen={true} title="Example">
      <FormField 
        label="Name" 
        type="text" 
        placeholder="Enter name" 
      />
    </BaseModal>
  );
}
```

### Using Hooks

```jsx
import { useFormData, useFileUpload } from '@/hooks';

function MyForm() {
  const { users, categories, loading } = useFormData();
  const fileUpload = useFileUpload({
    onSuccess: () => console.log('Upload complete!'),
    onError: (error) => console.error('Upload failed:', error)
  });
  
  // Component logic here
}
```

## üîß Development Guidelines

### Component Creation
When creating new components:

1. **Start with the story** - Define use cases first
2. **Follow naming conventions** - Use PascalCase for components
3. **Include prop validation** - Use PropTypes or TypeScript
4. **Add accessibility features** - ARIA labels, keyboard support
5. **Write comprehensive tests** - Unit tests for all variants
6. **Document thoroughly** - Include usage examples

### Code Standards
- **ESLint** configuration for consistent code style
- **Prettier** for automatic code formatting
- **JSDoc comments** for all functions and components
- **TypeScript** for type safety (where applicable)
